Concurrency Is Not Parallelism (And Why That Matters)

By: Abhishek

June 18, 2026

Almost everyone uses the words concurrency and parallelism as if they were the same thing. They are not. And the gap between them is exactly where most slow, buggy, hard-to-reason about programs are born.

This post builds the distinction up from the ground, one honest question at a time, until you can look at any piece of code and answer the only question that matters: is this dealing with lots of things at once, or is it doing lots of things at once? Those are not the same sentence.

1. What problem is concurrency actually solving?

Start with a program that does one thing: it reads a file, transforms it, and writes the result. It runs top to bottom. Nothing about it is confusing. Each line happens after the one before it.

Now give that program a job that involves waiting. Download a hundred web pages. Answer a thousand network requests. Read from ten slow disks. Suddenly your single line-after-line model is a disaster, because most of the time your program is just sitting there, blocked, waiting for something far away to respond.

That waiting is the whole reason concurrency exists. The processor is astonishingly fast; the world it talks to is astonishingly slow. A network round trip can cost as much time as millions of instructions. If your program freezes during every wait, it spends almost its entire life idle.

Picture it. You put a pot of water on to boil, and then stand there, staring at it, doing nothing else until it bubbles.

Or: you put the water on, and while it heats, you chop the vegetables, lay the table, and reply to a text. Same stove, same pair of hands, but nothing sits idle.

That second kitchen is concurrency: structuring the work so that when one task is stuck waiting, another gets to make progress.

Read that again, because it's the seed of everything below: concurrency is about structure, not speed. It's a way of organizing work so that "waiting" stops being wasted time.

2. Concurrency vs. parallelism, finally pinned down

Here is the cleanest way I know to hold the two ideas apart, borrowed from Rob Pike:

  • Concurrency is dealing with many things at once. It is a way to structure a program.
  • Parallelism is doing many things at once. It is a way to execute a program.

A program can be concurrent without ever being parallel. A single CPU core, switching rapidly between tasks, is concurrent: it is juggling many things, but at any single instant it is only doing one. A program can also be parallel without being especially "concurrent" in design, like a vector instruction adding eight numbers at once.

The practical punchline: concurrency is a property of your code; parallelism is a property of the machine running it. Good concurrent design lets the runtime add parallelism for free when the hardware allows it — but the two knobs are separate.

3. A coffee shop with one barista

Picture a coffee shop with exactly one barista. A customer orders a latte. The barista starts the espresso machine — and that machine takes thirty seconds, all on its own.

A sequential barista stands and stares at the machine for those thirty seconds, then takes the next order. The queue grows. Everyone is miserable.

A concurrent barista starts the espresso, and while it runs, takes the next order, steams milk for a third, rings up a fourth. One barista, one pair of hands — but no time wasted standing still. That is concurrency: one worker, many tasks in flight, progress whenever someone would otherwise be idle.

Parallelism is hiring a second barista. Now two drinks are genuinely being made at the same instant. Notice these are independent decisions: you can have the smart single barista (concurrent, not parallel), or two dumb baristas who each freeze at the machine (parallel, not concurrent), or — best of all — two smart baristas.

4. Threads, the OS, and the cost of switching

How does one CPU core "juggle"? The operating system gives each task a slice of time, then interrupts it, saves its state, and loads another's. This is a context switch, and it happens thousands of times a second, far faster than you can perceive. The illusion of "everything running at once" is built from very fast turn-taking.

But a context switch is not free. The OS has to save registers, swap stacks, and — most expensively — the new task often finds the CPU caches full of the old task's data, so it stalls waiting on memory. Switch too often and you spend more time switching than working. 

// Each OS thread carries real weight:
//   - its own stack (often ~1 MB reserved)
//   - kernel bookkeeping
//   - a context-switch cost when it's scheduled

// This is why spawning 100,000 OS threads is a bad idea,
// but spawning 100,000 lightweight tasks (goroutines,
// async tasks, green threads) is routine.

This is the gap modern runtimes exploit. Go's goroutines, Rust's async tasks, and Java's virtual threads are all cheap units of concurrency that the runtime multiplexes onto a small pool of real OS threads. You write "a million concurrent tasks," and the runtime quietly maps them onto the eight cores you actually have.

5. The reason shared state hurts

Everything above sounds wonderful. Here is where it gets dangerous: the moment two tasks touch the same piece of memory, the clean turn-taking model breaks, because a single line of your code is not a single step to the machine.

Take the most innocent statement in programming: 

counter = counter + 1;

To the CPU, that is three separate steps:

  • Read the current value of counter from memory into a register.
  • Add one to the value in the register.
  • Write the register back to counter in memory.

A context switch can happen between any of those steps. And that single fact is the source of an entire category of the nastiest bugs in software.

6. Race conditions you can see

Two tasks, A and B, both want to increment a shared counter that starts at 0. You expect the result to be 2. Watch what an unlucky interleaving does: 

counter = 0

Task A: read counter   -> sees 0
Task B: read counter   -> sees 0      (A hasn't written yet!)
Task A: add 1          -> 1
Task B: add 1          -> 1
Task A: write counter  -> counter = 1
Task B: write counter  -> counter = 1   // we lost an increment

Final value: 1   (expected 2)

Nobody wrote a bug. Each task ran correct code. The defect lives in the timing — in the assumption that "read, add, write" would happen without interruption. This is a race condition, and the cruelest thing about it is that it usually works. The bad interleaving might show up one time in a million, only under load, only in production, never on your laptop.

A race condition is a bug whose existence depends on luck. That is exactly what makes it so hard to find.

7. Locks, and the new problems they create

The classic fix is a lock (a mutex). Before touching shared state, a task must acquire the lock; everyone else waits. The protected region becomes mutually exclusive — only one task inside at a time. 

lock.acquire()
counter = counter + 1   // now safe: no one else can be here
lock.release()

This works, but you have traded one problem for several:

  • Contention. If everyone waits on the same lock, your carefully concurrent program serializes right back into a single-file line. You bought threads and then forbade them from running.
  • Deadlock. Task A holds lock 1 and wants lock 2; Task B holds lock 2 and wants lock 1. Both wait forever. Nothing crashes — it just silently stops.
  • Forgetting. Miss a single lock.acquire() on one code path and the race quietly returns, invisible until it isn't.

Locks are not wrong. But they put the entire burden of correctness on you remembering to do the right thing, everywhere, forever. There is a reason an entire branch of language design tries to make the compiler carry that burden instead.

8. Message passing: don't share, communicate

There is a different philosophy, summed up by Go's most quoted line:

Do not communicate by sharing memory; instead, share memory by communicating.

The idea: stop letting many tasks reach into the same variable. Instead, give the data one owner, and let other tasks send it messages through a channel. If only one task ever touches the counter, there is no race to protect against — the problem is designed out of existence rather than guarded against. 

// One task owns the counter. Others just send requests.
ch := make(chan int)

// owner:
go func() {
    count := 0
    for delta := range ch {   // process one message at a time
        count += delta
    }
}()

// anyone else:
ch <- 1   // "please add one" — no lock, no shared variable

Because the owning task handles one message at a time, the increments can never interleave. The same idea powers the actor model (Erlang, Akka) and is why those systems can run millions of concurrent entities without a mutex in sight.

9. So what should you actually reach for?

There is no universal winner, but there are good defaults:

  • Lots of waiting (network, disk, I/O): reach for async/await or lightweight tasks. The bottleneck is latency, not the CPU, so concurrency alone buys most of the win.
  • Lots of computation (image processing, simulation): reach for real parallelism — a thread pool across cores — because here the CPU is the bottleneck.
  • Shared mutable state is unavoidable: prefer message passing or immutable data first; use locks only when you must, and keep the locked region tiny.

And the meta-rule above all of them: the less state you share, the less can go wrong. Most concurrency pain is really just the pain of shared mutable state. Remove the sharing and most of the monsters vanish.

10. The whole story, in one breath

Programs wait, and waiting is wasted time. Concurrency is structuring code so that when one task is stuck, another can run — that's dealing with many things at once. Parallelism is the hardware actually doing many things at once. One is design, the other is execution, and you can have either without the other.

A single core fakes concurrency by switching tasks thousands of times a second, and that switching isn't free. The real danger starts when tasks share memory: one harmless-looking line is several machine steps, a switch can land in the middle, and you get a race condition — a bug that depends on luck. Locks fix it but invite contention and deadlock. Message passing sidesteps it by giving data a single owner.

So the next time you reach for a thread, ask the only question that matters: am I trying to deal with many things, or do many things? Answer that honestly, share as little as you can, and most concurrency becomes calm.
